# AGENTS.md

Instructions for AI coding agents (Claude Code, Codex, Copilot, Cursor, etc.) contributing to franktheunicorn.

## Before You Start

1. **Read `CLAUDE.md`** for full architecture context, project structure, and coding conventions. This file covers what's specific to AI agent contributions.
2. **Read the design doc** at `docs/franktheunicorn-master-design.md` if your task touches architecture or adds new features.
3. **Check the version roadmap** in CLAUDE.md. Do not implement features from a later version than the branch targets.

## PR Requirements

Every PR from an AI agent must include:

### In the PR description
- A clear one-paragraph summary of what changed and why
- Which design doc section the change relates to (e.g., "Implements §2.4 Auto-Detect Collaborators")
- Your session/task identifier so the operator can send feedback directly:
  ```
  Generated by: [agent name]
  Session: [session URL or task ID]
  ```
- If you're unsure about a design decision, say so explicitly in the PR description rather than guessing

### In the code
- Type hints on all public functions
- Docstrings on all public classes and non-trivial functions
- No `# TODO` without a linked issue number
- No commented-out code (remove it or don't write it)

### In the tests
- Tests for every new function and class
- Both API and scrape path tests if touching `data_access/`
- Factory fixtures via `factory_boy` — never raw `Model.objects.create()` in tests
- Run `pytest` locally before submitting. The PR will be rejected if coverage drops below 90%.

## What Will Get Your PR Rejected

These are learned from the operator's review patterns. If you produce findings matching these, the review agent will suppress them and the operator will reject the PR.

- **Introducing JavaScript frameworks.** No React, Vue, Svelte, Alpine. htmx + vanilla JS only.
- **Using `print()` instead of `logging`.** Always use the Django logger.
- **Postgres-specific SQL or ORM features.** SQLite must work. No `ArrayField`, no `JSONBContains` on SQLite-incompatible paths without a fallback.
- **Adding dependencies without justification.** Every new dependency is reviewed. Explain why it's needed in the PR description. Prefer stdlib.
- **Auto-posting to GitHub without the draft-only gate.** In v1, all comments go through the review queue. No shortcuts.
- **Mocking in a way that hides bugs.** Don't over-mock. If your test passes but would fail against a real API response, the mock is too permissive. Use VCR cassettes or saved fixtures from `review-agent record-fixtures`.
- **Huge PRs without test file changes.** If you're changing source code, add tests. If your tests pass on both PR and base branch, they don't test your change.
- **Modifying `~/.review-agent/` paths in tests.** Use `tmp_path` fixtures. Never touch the real config directory.

## Coding Patterns to Follow

### The DataFetcher pattern (dual-path)

If you're adding or modifying external data access, follow this pattern:

```python
# data_access/github/blame.py
from dataclasses import dataclass
from franktheunicorn.data_access.base import DataFetcher, FetchResult

@dataclass
class BlameLine:
    line_number: int
    author: str
    commit_sha: str
    content: str

@dataclass  
class BlameResult(FetchResult):
    file_path: str
    lines: list[BlameLine]

class GitHubBlameFetcherAPI(DataFetcher[BlameResult]):
    async def fetch_via_api(self, repo: str, ref: str, path: str) -> BlameResult:
        ...

class GitHubBlameFetcherScrape(DataFetcher[BlameResult]):
    async def fetch_via_scrape(self, repo: str, ref: str, path: str) -> BlameResult:
        ...
```

Then write contract tests:

```python
# tests/data_access/github/test_blame_contract.py
@pytest.mark.parametrize("fetcher_cls", [GitHubBlameFetcherAPI, GitHubBlameFetcherScrape])
def test_blame_returns_consistent_schema(fetcher_cls, mock_data):
    fetcher = fetcher_cls()
    result = fetcher.fetch(repo="apache/spark", ref="main", path="core/src/main.scala")
    assert isinstance(result, BlameResult)
    assert all(isinstance(l, BlameLine) for l in result.lines)
```

### The ReviewFinding pattern

When generating findings, always:
1. Check anti-patterns before surfacing
2. Run through Tone Guard
3. Set `sources` to identify which backend produced it
4. Include a `reasoning_trace`

```python
finding = ReviewFinding(
    pr=pr,
    file_path="src/main.py",
    line_start=42,
    line_end=42,
    category="correctness",
    severity="important",
    body="Use isNullAt(idx) instead of == null for Spark's internal null handling.",
    reasoning_trace="Line uses == null comparison on a Spark Row field. "
                    "Spark uses a special null representation that == null misses.",
    sources=["agent-primary"],
    confidence="high",
)
```

### Configuration access

Always access config through the loader, never read YAML directly:

```python
from franktheunicorn.config.loader import get_project_config, get_operator_config

project = get_project_config("apache-spark")
operator = get_operator_config()
```

## Commit Messages

Follow conventional commits:

```
feat(scoring): add blame-based proximity scoring (Layer 2)
fix(data_access): handle GitHub 403 rate limit response in scrape path  
test(review): add contract tests for tone guard rewriting
docs: update CLAUDE.md with new data source instructions
refactor(worker): extract priority queue into separate module
```

Always include the scope. Keep the subject under 72 characters.

## Working with the Test Suite

```bash
# Run everything (what CI runs)
pytest

# Run fast — skip slow integration tests
pytest -m "not slow"

# Run just the module you're working on
pytest tests/scoring/ -v

# Run with coverage report to find gaps
pytest --cov-report=html
open htmlcov/index.html

# Check types
mypy src/franktheunicorn/

# Check style
ruff check .
ruff format --check .
```

## When You're Stuck

- **Design question?** Check `docs/franktheunicorn-master-design.md`. The answer is probably there.
- **Unclear on a convention?** Check existing code in the same module for patterns.
- **Not sure if a feature belongs in this version?** Check the version roadmap in CLAUDE.md. When in doubt, leave it out and mention it in the PR description.
- **Hit a rate limit?** Check `review-agent check-rate-limits` and use the scrape path.
- **Test fixture out of date?** Run `review-agent record-fixtures --source github --project <project>` to regenerate.
